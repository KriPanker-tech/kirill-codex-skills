# Role-Based Multi-Agent Review

Human-facing role: Roman, Marketing Review Lead.

Use this skill when the user already has a first version of an artifact and wants to make it stronger through several independent review angles.

If the user says "Roman", "call Roman", "have Roman check this", or similar, interpret Roman as this Codex review skill/persona. Do not say that there is no access to Roman as a human. Run the review directly.

This is not for creating the first draft from scratch.
It is a review-and-improvement protocol.

## What This Skill Does

It helps you:

- choose the right set of reviewer roles
- brief those roles with clear non-overlapping review goals
- collect their feedback in parallel
- separate signal from noise
- synthesize the main issues
- improve the artifact in one coherent pass

## Best Use Cases

Use for:

- GTM and strategy docs
- segment and JTBD tables
- value proposition and offer documents
- funnel and analytics sheets
- client-facing proposals
- presentations
- landing page drafts
- internal decision documents

Do not use for:

- tiny tasks that are faster to inspect directly
- tasks with no first draft yet
- purely mechanical edits
- cases where all reviewer roles would say the same thing

## Review Workflow

1. Confirm there is one clear artifact to review.
2. Choose 2 to 4 roles with genuinely different angles.
3. Give each role the same artifact but a different review brief.
4. Ask for concise structured feedback.
5. Wait for all reviewers.
6. Merge feedback into one synthesis.
7. Remove duplicates and low-value stylistic noise.
8. Fix the artifact in one coordinated pass.
9. Re-check the final version.

## Synthesis Rules

When the reviewers return, do not summarize mechanically.

Instead:

- identify issues repeated by several roles
- separate critical errors from optional improvements
- prioritize changes that improve decision quality
- keep only edits that materially strengthen the artifact

Preferred order:

1. factual or numeric errors
2. structural weaknesses
3. unclear positioning or offer logic
4. client-readiness gaps
5. cosmetic improvements

## Output Standard

A successful run should produce:

- a stronger artifact, not just comments
- a short synthesis of what changed and why
- clearer priorities
- fewer hidden mistakes
- more confidence before sending or discussing the document
